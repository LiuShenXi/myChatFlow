# ChatFlow 多模态图片闭环设计

## 背景

当前项目已经具备部分图片相关能力：

- 输入区支持粘贴、拖拽和选择图片
- 前端会把图片压缩为 base64 并暂存到 `chat-store`
- 数据库 `Message` 已包含 `images String[]`

但整条多模态链路尚未打通：

- `useChat` 提交时没有把图片并入用户消息
- `/api/chat` 没有把图片转换成模型可消费的多模态内容块
- 历史消息加载与消息渲染仍基本只按纯文本处理
- `supportsVision` 仅存在于模型定义中，没有参与真实的产品行为

这导致界面“看起来可以上传图片”，但模型实际上并不能真正消费图片输入，属于能力未闭环状态。

## 目标

- 打通“图片上传 -> 消息提交 -> 模型消费 -> 消息持久化 -> 历史回显”的完整闭环
- 让支持视觉的内置模型真正支持文本加图片输入
- 为不支持视觉的内置模型提供清晰、可预期的阻止与提示
- 按产品规则让自定义模型默认允许图片输入
- 保持当前数据库结构可用，不在本次引入通用附件系统
- 在实现方式上为后续接入 RAG 预留清晰演进点

## 非目标

- 不在本次引入统一的附件/文件抽象
- 不做 PDF、网页、Markdown 等文档型 RAG
- 不实现模型输出图片、图片编辑或多媒体生成
- 不对自定义模型做自动多模态能力探测
- 不修改现有消息持久化 schema 为更复杂的 parts 结构

## 方案选择

### 方案 A：最小补丁

仅在发送前临时把图片拼进请求，尽快让模型能收到图片。

优点：

- 开发快
- 改动小

缺点：

- 历史回放、消息渲染、接口契约仍然割裂
- 后续还需要再补一轮整理

### 方案 B：闭环重接

保留现有数据库结构，在前端提交层与 `/api/chat` 之间增加“消息归一化层”，把文本和图片统一转换为模型输入消息，同时同步打通历史回显和模型能力约束。

优点：

- 一次补齐真实可用链路
- 与现有 schema 兼容
- 风险和收益平衡最好
- 后续可自然扩展到 RAG 场景

缺点：

- 会涉及输入区、聊天页、API、消息渲染和测试多层改动

### 方案 C：通用附件模型

直接引入更通用的 message parts / attachments 体系，为图片、文件、音频、检索块统一建模。

优点：

- 长期扩展性最好

缺点：

- 明显超出当前问题范围
- 实施成本高，容易把“补齐多模态”升级成底层重构

## 采用方案

采用方案 B：闭环重接。

同时按“向方案 C 演进的方案 B”实现，即：

- 本次不重做数据库层
- 本次新增一层专注于“业务消息 -> 模型输入消息”的归一化逻辑
- 后续如接入 RAG，可在这层继续扩展 retrieval context、citation、document chunks 等内容块

## 设计原则

- 优先补齐真实用户路径，而不是抽象未来全部附件场景
- 前后端都做约束，避免前端提示与后端行为不一致
- 持久化与模型输入解耦：数据库继续存 `content + images[]`，模型输入按需要转换为多模态内容块
- 用户已添加的图片不应因切换模型被静默丢弃
- 对自定义模型默认放行图片输入，但不额外承诺“已验证支持视觉”

## 数据与架构设计

### 持久化层

本次保持现有 `Message` schema 不变：

- `content String @db.Text`
- `images String[]`

其中：

- `content` 存储用户输入文本
- `images` 存储用户提交的图片 base64 数组

assistant 回复仍按当前方式只保存文本。

### 归一化层

新增一层消息归一化逻辑，职责是把业务消息转换成模型输入消息。建议放在独立模块，例如：

- `src/lib/chat/message-parts.ts`

该层至少承担以下职责：

- 把 UI / API 收到的用户消息转换成模型可消费的 content parts
- 根据当前模型判断是否允许图片输入
- 把数据库消息映射回前端可渲染的消息结构

这样可以避免把多模态判断散落在页面、API 路由和消息组件中。

### 前端提交模型

发送消息时，前端需要把：

- 输入框文本
- `chat-store` 中暂存的图片数组

合并为一条用户消息后再交给 `useChat` 提交。

本次不会让 `useChat` 直接管理图片状态，而是继续由 `chat-store` 持有图片，再在提交边界组装消息。

### 模型输入结构

对于纯文本消息：

- 继续按文本消息处理

对于带图的用户消息：

- 归一化为 AI SDK 可识别的多模态内容块
- 至少包含文本块与图片块

这层是未来承接 RAG 的关键扩展点。后续如果需要，可在同一层继续加入：

- retrieval context
- citations
- document chunks

## 模型能力规则

### 内置模型

内置模型继续以 `supportsVision` 作为是否支持图片输入的能力位。

规则如下：

- `supportsVision = true`：允许文本加图片发送
- `supportsVision = false`：若消息带图，则阻止发送并提示

### 自定义模型

按本次产品决策，自定义模型默认允许图片输入。

原因：

- `baseUrl + modelId` 难以可靠自动识别真实视觉能力
- 当前优先目标是保证多模态链路可用，而不是做能力探测

说明：

- “默认允许上传图片”不等于“系统已验证该模型支持视觉”
- 若具体自定义模型服务端不支持图片，错误仍由实际请求结果决定

## 前端交互设计

### 输入区

输入区保留当前图片上传方式：

- 点击选择图片
- 粘贴图片
- 拖拽图片

继续显示图片缩略图预览，并允许逐张删除。

发送成功后：

- 清空输入框中的图片预览

发送失败时：

- 保留已添加图片，避免用户重复上传

### 不支持视觉模型的行为

对于不支持视觉的内置模型：

- 不隐藏上传入口
- 不在切换模型时自动清除已上传图片
- 在用户发送时阻止提交，并给出明确提示

采用“发送时阻止”的原因：

- 不会静默删除用户内容
- 用户对错误更容易建立因果关系
- 切回支持视觉模型后，原图片仍可继续发送

### 模型选择器

可为支持视觉的内置模型补充轻量能力标识，例如：

- “视觉”标签
- 图片图标

自定义模型不展示“已支持视觉”标签，避免误导。

## 历史消息与渲染设计

### 历史加载

加载会话历史时，如果用户消息包含 `images[]`：

- 需要把图片映射回前端消息结构
- 刷新页面后仍可看到图片缩略图

### 消息渲染

用户消息：

- 显示文本内容
- 若存在图片，则在消息气泡中或相邻区域显示缩略图

assistant 消息：

- 继续沿用现有 Markdown 文本渲染

本次不引入 assistant 图片输出展示。

## API 行为设计

### `/api/chat`

新增如下处理逻辑：

1. 解析当前模型
2. 判断该模型是否允许图片输入
3. 若消息带图且模型不支持视觉，则返回明确 `400`
4. 若允许，则将消息归一化为模型输入结构
5. 调用 `streamText`
6. `onFinish` 时继续持久化：
   - 用户消息 `content`
   - 用户消息 `images[]`
   - assistant 文本回复

### 错误语义

本次新增的核心错误语义为：

- 当消息包含图片，但当前模型不支持视觉时，返回 `400`
- 错误文案建议统一为：`当前模型不支持图片输入，请切换到支持多模态的模型后再发送`

其他错误语义保持现状：

- 未登录：`401`
- 模型不存在或越权：`400`
- 自定义模型缺少 API Key：保持现有错误提示

## 测试策略

### 1. 聊天页与输入区

覆盖：

- 提交时会把文本和图片组装为一条用户消息
- 成功发送后会清空图片
- 发送失败时不会清空图片
- 不支持视觉的模型带图发送时会阻止提交并提示

### 2. `/api/chat` 路由

覆盖：

- 支持视觉的内置模型会把图片消息归一化后传给 `streamText`
- 不支持视觉的内置模型带图请求会返回 `400`
- 自定义模型带图请求允许通过
- `onFinish` 仍正确持久化 `content + images[]`

### 3. 历史消息加载

覆盖：

- 会话历史中的用户图片消息会被正确映射回前端消息结构

### 4. 消息渲染

覆盖：

- 纯文本用户消息正常显示
- 带图片的用户消息会显示缩略图
- assistant Markdown 渲染不受影响

## 与后续 RAG 的衔接

本次不直接实现 RAG，但要为其保留自然演进点。

约束如下：

- 不把图片硬编码进多个页面与路由分支
- 把“模型输入归一化”集中在单独模块
- 未来新增 RAG 时，可在该层继续引入：
  - 文本上下文块
  - 检索命中块
  - 引用标记
  - 文档片段

因此，本次方案的定位是：

- 不做通用附件系统
- 但为后续 `text + image + retrieval context` 的输入组合打基础

## 风险与权衡

### 风险 1：自定义模型默认允许图片，实际网关可能不支持

接受该风险。

原因：

- 这是当前产品决策
- 自动探测成本高且可靠性差
- 即使失败，也应由具体模型服务返回真实错误

### 风险 2：base64 图片进入消息链路，可能增加请求体体积

当前接受该风险。

原因：

- 项目现阶段已采用 base64 图片存储方案
- 本次目标是打通功能闭环，不在此时切换对象存储方案

### 风险 3：`useChat` 与自定义消息结构的边界处理不当

通过测试和独立归一化层降低风险。

## 验收标准

- 支持视觉的内置模型可发送“文本 + 图片”并得到响应
- 不支持视觉的内置模型在带图发送时会被明确阻止
- 自定义模型默认允许图片发送
- 用户历史消息刷新后仍可看到已发送图片
- 现有纯文本聊天路径不回归
- 当前实现不阻塞未来接入 RAG
