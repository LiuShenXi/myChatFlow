# ChatFlow 登录后与会话切换后的输入框自动聚焦设计

## 背景

当前 ChatFlow 已经补齐了未登录提示、自动创建首条会话，以及“进入系统后尽快达到可输入状态”的主流程，但仍有一个明显的体验断点：

- 用户登录完成后，虽然首页已经可用，但通常还需要手动点一次输入框才能开始输入
- 页面刷新后，即使会话和模型状态已经恢复，焦点也不会自动回到聊天输入框
- 切换到新会话后，输入上下文已经变化，但输入焦点不会自动跟随

这不会阻塞功能使用，但会让“登录即聊 / 切换即聊”的体验不够顺滑。用户明确希望：

1. 登录后触发自动聚焦
2. 刷新页面重新进入可输入状态后触发自动聚焦
3. 切换到新会话后触发自动聚焦
4. 但如果用户已经在其他输入控件里操作，不要强行抢走焦点

## 目标

1. 在聊天页重新进入“可输入状态”时，自动将焦点移动到聊天输入框
2. 覆盖登录后、刷新后、切换会话后三类典型时机
3. 避免打断用户在其他输入控件中的操作
4. 保持现有会话初始化、流式响应和未登录提示逻辑不变

## 非目标

- 不新增全局快捷键系统
- 不新增 store 级“待聚焦”事件或信号
- 不改动 API、数据库结构或会话数据模型
- 不在未登录、流式响应中、或无当前会话时强制聚焦

## 方案比较

### 方案 A：聊天页作为聚焦协调器

由聊天页统一判断当前是否“进入可输入状态”，并在满足条件时通过输入框 ref 调用 `focus()`。

优点：

- 触发规则集中，容易覆盖登录后、刷新后、切换会话后三种情况
- 输入框组件仍保持简单，只负责输入本身
- 更容易实现“只在状态变化时尝试一次”的控制

缺点：

- 需要让 `InputArea` 暴露内部 `textarea` ref

### 方案 B：InputArea 自己决定何时聚焦

让 `InputArea` 根据 `disabled`、登录态和会话状态自己判断何时调用 `focus()`。

优点：

- 代码改动表面上较少

缺点：

- 组件会知道太多页面语义
- 容易把“是否可输入”和“是否应该抢焦点”耦合在一起
- 后续增加新的触发条件时更难维护

### 方案 C：Store 驱动聚焦信号

在 store 中增加 `shouldFocusInput` 或自增 token，由登录完成、会话切换等流程发出一次聚焦信号，聊天页消费后执行 `focus()`。

优点：

- 触发来源形式上比较清晰

缺点：

- 为一个局部体验细节增加跨组件状态同步
- 会让 store 承担本不属于状态管理层的瞬时 UI 副作用

## 采用方案

采用方案 A：由聊天页作为页面级聚焦协调器，`InputArea` 仅暴露 `textarea` 引用，不承载聚焦策略。

## 设计细节

### 1. “进入可输入状态”的定义

聊天页只在以下条件同时满足时，才认为输入框处于可聚焦的可用状态：

- `status === "authenticated"`
- `currentSessionId` 存在
- 当前不在流式响应中，即输入框不是 `disabled`
- 当前页面没有其他输入型焦点需要保留

其中“没有其他输入型焦点”是一个保护性前置条件，不参与 UI 可用性计算，但参与是否执行自动聚焦。

### 2. 触发时机

本期自动聚焦覆盖三类触发：

- 登录后首次进入首页并完成会话准备
- 页面刷新后重新恢复到可输入状态
- 切换到新的当前会话后，页面重新进入可输入状态

实现上不依赖“这是哪一种场景”的业务标签，而是统一收敛为两类状态变化：

- 从“不可输入”切换到“可输入”
- `currentSessionId` 发生变化，且新会话已进入可输入状态

这样可以避免为登录、刷新、切换会话分别维护三套逻辑。

### 3. 防打扰规则

在真正调用聊天输入框的 `focus()` 之前，先读取 `document.activeElement` 并判断当前焦点是否已经位于以下元素之一：

- `input`
- `textarea`
- `select`
- 带 `contenteditable="true"` 的元素

如果当前已有输入型焦点，则本次自动聚焦跳过，不抢焦点。

这条规则意味着：

- 用户正在设置弹窗里输入 API Key 时，不会被聊天框打断
- 用户正在登录页或未来的其他表单控件里输入时，不会被抢焦点
- 用户只是点击了页面空白区域或普通按钮时，聊天页仍可以把焦点引回输入框

### 4. 组件边界

#### `src/components/chat/InputArea.tsx`

职责保持不变，继续负责：

- 文本输入
- 图片上传
- 提交消息
- 自适应高度

本期只新增一个能力：向外暴露内部 `Textarea` 对应的 DOM 引用，供聊天页在合适时机调用 `focus()`。

建议做法：

- 使用 `forwardRef<HTMLTextAreaElement, InputAreaProps>`，把内部 `textareaRef` 同步给外部
- 或在 props 中接收 `textareaRef`

两种都可行，本期更推荐 `forwardRef`，因为语义更清楚，也更符合“组件暴露聚焦目标”的模式。

#### `src/app/(chat)/page.tsx`

新增页面级聚焦协调 effect，负责：

- 维护输入框 ref
- 判断当前是否进入可输入状态
- 比较上一次是否已经处于可输入状态
- 识别 `currentSessionId` 是否发生变化
- 在满足条件且不打扰其他输入的前提下调用 `focus()`

页面层不需要新增 store 字段，只需利用已有的：

- `status`
- `currentSessionId`
- `isLoading`

以及一个局部 `useRef` 用来记录上一次的可输入状态和会话 ID。

### 5. 状态流

推荐的判断顺序如下：

1. 先根据 `status`、`currentSessionId`、`isLoading` 计算当前是否可输入
2. 再对比上一轮是否可输入，以及 `currentSessionId` 是否变化
3. 若两者都不满足触发条件，则不做任何事
4. 若满足触发条件，则检查 `document.activeElement`
5. 若当前焦点不是输入型元素，则聚焦聊天输入框
6. 最后更新“上一次可输入状态”和“上一次会话 ID”的记录

这样可以避免：

- 首屏每次 render 都重复 `focus()`
- 流式响应开始/结束时频繁误触发
- 同一会话下的无关重渲染重复抢焦点

### 6. 错误处理与兼容性

本期不需要额外错误提示，但应保留以下保护：

- 如果输入框 ref 为空，则静默跳过
- 如果页面运行环境不是浏览器，不访问 `document`
- 如果 `focus()` 调用失败，不抛出额外 UI 错误

本功能属于增强体验，不应该影响现有对话流程和渲染稳定性。

## 测试策略

采用 TDD，重点覆盖聊天页的页面级行为，而不是为这个细节引入大量组件级测试。

### 1. `__tests__/app/chat-page.test.tsx`

新增或扩展以下用例：

- 当页面进入可输入状态且当前没有输入型焦点时，会尝试聚焦聊天输入框
- 当用户未登录时，不会触发自动聚焦
- 当没有当前会话时，不会触发自动聚焦
- 当当前焦点已经在其他输入控件中时，不会抢焦点
- 当 `currentSessionId` 从一个值切换到另一个值时，会再次尝试聚焦新会话输入框

测试实现建议：

- 在 `InputArea` 的 mock 中暴露一个可观测的 `focus` 行为
- 使用 `document.body.appendChild()` 创建额外 `input` 或 `textarea`，模拟“已有输入型焦点”
- 使用 `rerender()` 或调整 store state，验证会话切换后的再次触发

### 2. 可选组件级验证

如有需要，可为 `InputArea` 增加一个极小测试，确认外部 ref 最终指向真实 `textarea`。但这不是本期主验证重点，优先保证聊天页行为测试完整即可。

## 验收标准

完成后应满足：

1. 登录后进入首页并准备好首条可用会话时，聊天输入框会自动聚焦
2. 刷新页面后，只要恢复到可输入状态，聊天输入框会自动聚焦
3. 切换到新会话后，只要该会话已进入可输入状态，聊天输入框会自动聚焦
4. 如果用户当前已经在其他输入控件里输入，不会被聊天输入框抢走焦点
5. `npm run verify:local` 通过
