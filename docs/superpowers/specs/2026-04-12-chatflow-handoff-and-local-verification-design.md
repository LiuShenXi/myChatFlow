# ChatFlow 收工交接与本地联调收口设计

## 背景

当前 `myChatFlow` 主线功能已经完成到可交付的基础状态：

- Task 11 到 Task 14 的主体代码已完成并合并到 `main`
- 自动化验证已跑通：
  - `npx tsc --noEmit`
  - `npm test -- --runInBand`
  - `npm run lint`
  - `npm run build`
- 当前真正未闭环的是“真实环境联调”，而不是核心功能开发

也就是说，项目现在的主要阻塞项不是代码逻辑，而是以下运行条件尚未落地：

- 本地数据库连接未配置
- OAuth Provider 未配置
- `ENCRYPTION_KEY` 等运行时密钥未配置
- 尚未形成统一的本地联调入口和浏览器验收步骤

这次收工的目标不是继续扩功能，而是把“下次如何继续工作”明确沉淀下来，让下一次新对话可以直接进入执行。

## 本次收工目标

将下一阶段工作收敛为一个清晰、低风险、可快速启动的子项目：

**目标：**
补齐本地联调配套能力，使开发者在具备环境变量后，可以快速判断环境是否可用、按文档启动项目，并按清单完成真实环境验收。

**非目标：**

- 不新增聊天业务能力
- 不重构现有 UI/Store/API 架构
- 不调整现有数据库模型
- 不扩展新的模型提供商或消息能力

## 推荐推进方向

下一次开发建议只做“联调收口包”，包含四个交付物：

### 1. 环境自检入口

提供一个本地命令，用来检查联调所需的关键条件是否齐全，最少覆盖：

- `POSTGRES_PRISMA_URL`
- `POSTGRES_URL_NON_POOLING`
- `NEXTAUTH_URL`
- `NEXTAUTH_SECRET`
- `GOOGLE_CLIENT_ID`
- `GOOGLE_CLIENT_SECRET`
- `GITHUB_CLIENT_ID`
- `GITHUB_CLIENT_SECRET`
- `ENCRYPTION_KEY`

自检规则建议分为两类：

- 存在性检查：变量是否定义
- 轻量格式检查：例如 `ENCRYPTION_KEY` 是否为 64 位十六进制字符串，`NEXTAUTH_URL` 是否像 URL

自检输出目标：

- 明确列出缺失项
- 明确区分“阻塞启动”和“阻塞真实联调”的问题
- 给出下一步建议动作

### 2. 统一本地验证命令

补一个串行执行的本地验证命令，避免手动重复输入，也避免并行命令互相干扰构建产物。

建议形成两类命令：

- `doctor`：检查环境条件
- `verify:local`：串行执行代码验证

`verify:local` 目标流程：

1. `npx tsc --noEmit`
2. `npm test -- --runInBand`
3. `npm run lint`
4. `npm run build`

约束：

- 必须串行执行
- 输出需足够清晰，便于定位失败步骤

### 3. 本地联调文档

整理一份“下次打开项目就能照做”的联调说明文档，内容应以执行为导向，而不是泛泛介绍。

文档至少需要回答：

- 第一次进入仓库先看什么
- 本地需要准备哪些环境变量
- OAuth 回调地址如何配置
- 数据库最少要满足什么前置条件
- Prisma 相关命令如何执行
- 启动项目前的推荐顺序是什么
- 常见报错如何快速判断归属

建议把文档结构写成以下顺序：

1. 当前项目状态
2. 开始前检查
3. 环境变量准备
4. 数据库准备
5. OAuth 准备
6. 启动项目
7. 浏览器联调路径
8. 常见问题排查

### 4. 浏览器联调验收清单

把剩余真实环境验收步骤整理成 checklist，供下次直接执行。

建议清单项包括：

- 登录页显示正常
- Google 登录正常
- GitHub 登录正常
- 创建新对话
- 保存 OpenAI Key
- 保存 Anthropic Key
- 保存 DeepSeek Key
- 模型切换后发消息
- 会话切换
- 会话重命名
- 会话删除
- 退出登录

如果真实环境不足以覆盖所有 Provider，也要在清单里写明：

- 哪些项是“可选验证”
- 哪些项是“最低通过标准”

## 设计原则

### 1. 优先降低下次启动成本

下次进入项目时，不应该先花时间重新理解：

- 现在做到哪了
- 还差什么环境
- 应该先跑什么命令

这些信息应直接由文档和脚本给出。

### 2. 优先收口，不再扩需求

这个阶段的所有改动都应服务于“更快联调”，而不是“更多功能”。

判断标准：

- 不能帮助下次启动工作的内容，不做
- 不能帮助环境排查的内容，不做
- 会引入新的业务分支和维护成本的内容，不做

### 3. 保持低耦合

脚本、文档、验证命令应尽量独立于现有业务逻辑。

例如：

- 不为环境检查修改 API 行为
- 不为了文档而重构组件
- 不为了联调脚本而改现有业务文件结构

## 数据流与执行路径

下一次开发的理想执行路径应是：

1. 阅读交接文档，确认当前项目状态
2. 复制环境变量模板并填入真实值
3. 运行环境自检命令
4. 初始化 Prisma / 数据库
5. 启动开发服务器
6. 按浏览器联调清单逐项验证
7. 根据真实联调结果决定是否进入新的功能开发

## 错误处理与排查策略

联调收口阶段建议把问题分类记录：

### 环境类

- 缺失环境变量
- URL/密钥格式不正确
- OAuth 回调地址未配置
- 数据库连接失败

### 构建类

- Prisma Client 未生成
- `next build` 失败
- TypeScript 校验失败

### 运行类

- 未登录无法访问 API
- 登录后 session 不完整
- API Key 保存失败
- 模型切换后聊天调用失败

文档中应该明确：

- 每一类问题先看哪里
- 优先排查哪些文件/命令

## 验收标准

下一次执行完这部分工作后，应满足以下标准：

1. 新进入项目的人只看文档，就知道如何开始
2. 运行一个命令，就能知道本地环境是否具备联调条件
3. 运行一个命令，就能完成当前仓库的本地代码验证
4. 填好真实环境后，可以直接按清单完成浏览器联调

## 下一次开发建议起手顺序

下次新对话建议按以下顺序开始：

1. 先阅读本文件
2. 再确认 `main` 当前提交状态
3. 再检查是否存在 `.env.local`
4. 若无真实环境，则优先实现“环境自检脚本 + 联调文档 + 验收清单”
5. 若真实环境已具备，则先跑浏览器联调，再决定是否继续新功能

## 结论

当前项目最合理的下一步不是继续扩展功能，而是完成“本地联调收口包”。

这样做的收益最大：

- 能把已经完成的代码真正转化为可验证成果
- 能显著降低下次重新进入项目的成本
- 能为下一轮功能开发提供更稳定的起点

这将是一个边界清晰、风险可控、适合作为下一次开发开场任务的小型子项目。
