# 自定义模型视觉能力自动识别与失败学习设计

## 背景

当前项目对内置模型使用静态 `supportsVision` 能力判断，这对已知模型有效；但自定义模型的能力未知，过去一度采用“全部视为支持图片输入”的策略，导致文本模型在携带图片时被错误放行到上游接口。

在实际联调中：

- 自定义模型 `GLM5.1` 使用 `glm-5.1` 时，纯文本对话可工作
- 同一配置携带图片时，上游返回 `400 API 调用参数有误`
- 这类失败并不是临时网络问题，而是模型能力不匹配

如果我们继续对所有自定义模型一刀切：

- 全部允许图片：会持续把图片错误发给文本模型
- 全部禁止图片：会让视觉模型也无法使用

因此需要一种既不一刀切、又可逐步学习的能力识别方案。

## 目标

1. 为自定义模型引入“未知 / 视觉 / 非视觉”三态能力模型
2. 对未知模型默认允许尝试图片请求，降低首次使用门槛
3. 允许用户手动声明某条自定义模型支持图片输入
4. 当真实调用返回“明确不支持视觉”的证据时，系统自动学习并记录为非视觉模型
5. 自动学习后给用户一个清晰提示，而不是静默降级
6. 保留用户手动纠正能力，避免一次误判永久锁死

## 非目标

1. 不依赖额外 AI 服务来判断模型能力
2. 不尝试做跨所有 provider 的动态模型发现系统
3. 不在本次实现里为所有厂商接入官方模型列表 API
4. 不自动把一次成功的纯文本请求推断为“支持视觉”

## 方案概览

采用三层能力判断：

1. 显式能力字段：最高优先级
2. provider 已知规则推断：第二优先级
3. 运行时兜底：无法确认时按未知处理

同时引入“失败学习”：

- 未知模型默认允许尝试一次真实图片请求
- 只有命中“明确不支持视觉”的错误信号时，才自动标记为 `text-only`
- 自动标记后，后续请求直接按非视觉模型处理
- UI 明确提示用户：系统已自动识别该模型暂不支持图片输入，并已关闭图片能力

## 数据模型设计

为 `CustomModelConfig` 新增两个字段：

- `visionCapability`
  - 类型：枚举字符串
  - 值：`unknown` | `vision` | `text-only`
- `visionCapabilitySource`
  - 类型：枚举字符串
  - 值：`manual` | `inferred` | `learned`

字段语义：

- `unknown`
  - 系统暂时不能确认该模型是否支持图片
- `vision`
  - 该模型被认为支持图片输入
- `text-only`
  - 该模型被认为不支持图片输入

来源语义：

- `manual`
  - 用户在设置页主动选择
- `inferred`
  - 系统根据已知 provider 规则推断
- `learned`
  - 系统从真实失败中学习得到

默认值：

- `visionCapability = unknown`
- `visionCapabilitySource = inferred`

之所以默认 source 不是 `manual`，是为了区分“用户明确设置过”与“系统初始状态”。

## 自动识别策略

### 1. 显式字段优先

如果自定义模型配置中已有 `visionCapability`：

- `vision`：允许图片
- `text-only`：禁止图片
- `unknown`：允许先尝试真实图片请求，并在失败后学习

### 2. provider 已知规则推断

对能稳定识别的 provider，引入独立的规则函数，例如：

- 智谱 GLM
  - `glm-5.1`、`glm-5`、`glm-4.7` 视为文本
  - `glm-5v-turbo`、`glm-4.6v` 等视觉模型视为视觉

推断原则：

- 只使用“高置信度、稳定”的规则
- 不依赖脆弱的模糊正则，如仅凭名称里有 `v`、`vision`
- 规则命中后，更新配置为对应能力，来源标记为 `inferred`

### 3. 运行时兜底

若既没有手动声明，也没有命中 provider 规则，则视为 `unknown`。

`unknown` 的行为：

- 默认允许图片提交，避免把可用的视觉模型提前挡掉
- UI 可提示“当前模型图片能力尚未验证，若失败会自动关闭图片能力”
- 若真实请求命中高置信度“非视觉模型”错误，则自动学习为 `text-only`

## 失败学习策略

### 学习前提

只有在以下情况下才允许进入失败学习：

1. 当前自定义模型被设置为 `vision`，或当前能力为 `unknown` 且允许图片发送
2. 本次请求实际包含图片
3. 上游返回可明确归因为“视觉能力不支持”的错误

### 可学习的错误信号

只对“明确指向视觉不支持”的错误自动学习为 `text-only`。

例如：

- provider 明确错误码
- 明确文案：
  - 不支持图片输入
  - 仅支持纯文本
  - image_url / image content 非法且上下文明确指向模型不支持视觉

### 不可学习的错误信号

以下错误绝不能自动打上 `text-only` 标签：

- `401` / API Key 错误
- `429` / 限流
- 网络错误 / 超时
- 服务端 `5xx`
- 泛化的参数错误，但没有足够证据证明是“视觉不支持”

这一步是为了避免把暂时性问题误学习成模型能力问题。

### 学习结果

命中可学习错误后：

1. 后端更新该 `CustomModelConfig.visionCapability = text-only`
2. `visionCapabilitySource = learned`
3. 当前请求返回明确错误消息
4. 响应中携带一个可识别的提示标记，供前端展示 toast 或内联提示

建议提示文案：

`已自动识别该模型暂不支持图片输入，系统已为你关闭图片能力。`

## 前端交互设计

### 自定义模型设置页

在每条自定义模型配置上新增“图片能力”设置：

- 自动识别中
- 支持图片输入
- 不支持图片输入

展示建议：

- 一个能力状态标签
- 一段来源说明
  - 手动设置
  - 系统识别
  - 失败学习

可编辑行为：

- 用户可手动切换到“支持图片输入”
- 用户可手动切换到“不支持图片输入”
- 用户也可切回“自动识别”

这样即使系统误判，用户也能自行纠正。

### 聊天输入区

当当前模型为：

- `vision`
  - 正常允许图片上传和发送
- `text-only`
  - 阻止图片提交，显示“当前模型不支持图片输入”
- `unknown`
  - 默认允许图片提交
  - 可展示弱提示，说明系统会先尝试发送；如果模型不支持，会自动关闭图片能力

如果希望进一步优化体验，可以在未来加一个快捷入口，例如：

- “当前模型能力未知，仍要尝试发送图片吗？”

但本次不做额外交互弹窗，只保留非阻断提示。

## 后端数据流

### 请求前判断

`/api/chat` 接收到图片消息时：

1. 若内置模型，继续使用静态 `supportsVision`
2. 若自定义模型：
   - 读取 `visionCapability`
   - 若为 `text-only`，直接返回 400
   - 若为 `unknown`，允许继续调用
   - 若为 `vision`，允许继续调用

### 调用后学习

若请求已被允许并进入上游调用，但上游失败：

1. 解析 provider / 错误码 / 文案
2. 判断是否命中“明确不支持视觉”
3. 若命中：
   - 更新该模型能力为 `text-only`
   - 返回学习提示
4. 若未命中：
   - 按原错误链路返回，不更新能力

## 错误处理设计

新增一层“视觉能力错误分类器”：

- 输入：
  - provider
  - modelId
  - error
  - 当前请求是否包含图片
- 输出：
  - `not-vision-supported`
  - `unknown`

这个分类器只做能力判断，不负责直接拼 UI 文案。

然后在现有错误映射层上增加一种业务错误：

- `AUTO_LEARNED_TEXT_ONLY`

对应用户可见提示：

- 主错误：`当前模型不支持图片输入，请切换到支持多模态的模型后再发送`
- 附加提示：`已自动识别该模型暂不支持图片输入，系统已为你关闭图片能力`

## 为什么不用外部接口直接判断

理论上可以接入 provider 的模型列表或模型详情接口，但本项目当前不应把这件事建立在一个“不确定是否稳定存在”的外部元数据接口上，原因是：

1. 不同 provider 的能力元数据格式不统一
2. OpenAI-compatible 接口并没有统一标准字段表达 `supportsVision`
3. 增加运行时远程探测会引入延迟、失败面和缓存复杂度
4. 某些错误仍需要真实调用后才能确认

因此本次设计选择：

- 先用本地能力状态做真值管理
- provider 已知规则只作为辅助推断
- 失败学习作为动态纠偏机制

这是更稳、更可控的实现方式。

## 实现拆分建议

### 第一步：数据模型与 DTO

- 为 `CustomModelConfig` 增加能力字段
- API 返回值包含能力与来源

### 第二步：能力解析层

- 新增 `custom-model-capabilities.ts`
- 提供：
  - 读取最终能力
  - provider 推断
  - 失败学习判断

### 第三步：聊天请求接入

- `/api/chat` 在图片提交前读取最终能力
- 若上游失败，执行失败学习

### 第四步：设置页展示与编辑

- `CustomModelManager` 增加能力展示与编辑控件

### 第五步：输入区文案优化

- 对 `unknown` 与 `text-only` 区分提示
- 接收后端返回的自动学习提示

## 风险与权衡

### 风险 1：错误分类误判

如果 provider 报错文案变化，可能误判或漏判。

缓解：

- 分类逻辑按 provider 分层实现
- 仅对高置信度信号自动学习
- 保留用户手动改回能力

### 风险 2：未知模型第一次可能触发真实失败

默认放行会让一部分未知文本模型第一次真正打到上游，用户可能先看到一次失败。

缓解：

- 仅对高置信度“非视觉”错误自动学习
- 学习后自动关闭图片能力，避免重复踩坑
- 设置页保留手动改回能力的入口

### 风险 3：能力状态与真实模型版本漂移

同一模型名未来可能升级能力。

缓解：

- 用户可手动切回 `unknown` 或 `vision`
- 系统不把一次成功的文本请求视为“永久真相”

## 推荐结论

采用“显式字段 + provider 推断 + 运行时兜底 + 失败学习”的组合方案。

这是当前项目里最优雅、最稳妥的路径：

- 不靠 AI 做黑盒判断
- 不依赖不稳定的外部能力接口
- 不再对自定义模型一刀切
- `unknown` 先放行，兼顾首用体验
- 还能通过真实失败逐步积累能力认知

相对于“全放开”或“全禁用”，这个方案在准确性、可解释性和可恢复性上更平衡。
