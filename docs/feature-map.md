# ChatFlow 功能地图

## 认证与访问控制

- 页面可以在未登录状态打开，但核心 API 会按登录态做保护。
- `/api/sessions`、`/api/chat`、`/api/keys`、`/api/custom-models` 都要求用户已登录。
- 未登录时，用户只能浏览壳层界面，无法读取会话、保存密钥或发送消息。

## 会话能力

- 登录后会自动拉取当前用户的会话列表。
- 会话列表为空时，系统会自动创建首条会话。
- 支持会话切换、创建、重命名、删除。
- 会话会按最近更新时间排序，当前会话会随列表和本地状态同步切换。

## 聊天能力

- 支持流式文本返回。
- 支持 Markdown 渲染与代码高亮。
- 消息支持携带图片附件，前端会在提交前做能力校验。

## Provider 与模型配置

- 当前已支持 `OpenAI`、`Anthropic`、`DeepSeek`、`Qwen`、`GLM`、`Kimi`、`豆包`。
- API Key 会保存在服务端，刷新页面后仍能恢复已配置状态。
- 不同模型会自动对应到各自的 provider 路径，用户只需要在设置里完成一次配置。
- `豆包` 除了 API Key 之外还需要补充额外的接入参数。

## 自定义模型能力

- 已支持创建、编辑、删除自定义模型配置。
- 每条自定义模型都可以单独配置接入地址、模型标识和密钥。
- 自定义模型适合接入兼容 OpenAI 接口的外部模型服务。
- 系统会尽量识别自定义模型是否支持图片输入，并把结果记住，减少重复试错。
- 如果某个自定义模型多次返回不支持图片输入的结果，后续会被当作纯文本模型处理。

## 多模态能力边界

- 仓库已经具备图片上传与图片消息处理能力，所以不能把项目写成“完全不支持图片”。
- 内置模型会先按已知能力判断，支持图片的模型可直接进入多模态对话，不支持的模型会在发送前被拦住。
- 自定义模型在能力未知时通常会先尝试调用；如果实际反馈表明不支持图片，系统会记住这个结果，后续按纯文本模型处理。
- 图片能力的可用性最终仍受具体模型与 provider 限制，文档应把它描述成“可用边界”，而不是“统一开关”。

## 当前限制

- 未登录状态下不能读取会话、保存密钥或发送消息。
- 静态模型需要先在设置页配置对应 provider 的 API Key，否则无法聊天。
- 某些 provider 还会有额外接入参数要求。
- 自定义模型目前主要覆盖兼容 OpenAI 接口的接入方式，不等同于任意私有协议都能直接接入。
- 多模态能力并非全量统一，部分模型仍只能处理纯文本，图片可用性取决于模型本身和接入方能力。
