# ChatFlow 功能地图

## 认证与访问控制

- 页面可以在未登录状态打开，但核心 API 会按登录态做保护。
- `/api/sessions`、`/api/chat`、`/api/keys`、`/api/custom-models` 都要求用户已登录。
- 未登录时，界面会明确提示“登录后才能查看和创建会话”“登录后即可开始对话”“登录后才能配置 API Key”。

## 会话能力

- 登录后会自动拉取当前用户的会话列表。
- 会话列表为空时，系统会自动创建首条会话。
- 支持会话切换、创建、重命名、删除。
- 会话会按最近更新时间排序，当前会话会随列表和本地状态同步切换。

## 聊天能力

- 支持流式文本返回。
- 支持 Markdown 渲染与代码高亮。
- 发送消息时会把当前会话 ID 和当前模型一并带入聊天请求。
- 消息支持携带图片附件，前端会在提交前做能力校验。

## Provider 与模型配置

- 当前已支持 `OpenAI`、`Anthropic`、`DeepSeek`、`Qwen`、`GLM`、`Kimi`、`豆包`。
- API Key 由服务端加密持久化，刷新后仍可重新读取配置状态。
- 模型选择器会按模型元数据决定 provider 路由。
- `Anthropic` 保持独立实现，其余 provider 走 OpenAI-compatible 路由。
- `豆包` 额外依赖 `endpoint-id`，保存和发起对话时都会校验这一项。

## 自定义模型能力

- 已支持创建、编辑、删除自定义模型配置。
- 每条自定义模型会绑定自己的 `Base URL`、`Model ID` 和 `API Key`。
- 自定义模型走 OpenAI-compatible 调用方式。
- 自定义模型会记录图片能力状态，并支持手动设置、系统推断和失败学习三种来源。
- 如果自定义模型在图片对话中明确表现为不支持多模态，系统会自动标记为纯文本模型。

## 多模态能力边界

- 仓库已经存在图片上传、附件解析和能力识别相关实现。
- 不是所有模型都支持图片输入，前端和后端都会先做能力判断。
- 静态模型是否支持图片输入，由 `supportsVision` 决定。
- 自定义模型会结合 `visionCapability`、历史识别结果和实际调用反馈进行判断。
- 当前 README 不应再写成“完全不支持图片”，更准确的说法是“支持图片输入能力，但是否可用取决于具体模型与 provider”。

## 当前限制

- 未登录状态下只能看见提示，不能读取会话、保存 Key 或发送消息。
- 静态模型需要先在设置页配置对应 provider 的 API Key，否则无法聊天。
- `豆包` 除 API Key 外还必须配置 `endpoint-id`。
- 自定义模型目前只覆盖 OpenAI-compatible 接入方式，不等同于任意私有协议都可直接接入。
- 多模态能力并非全量统一，部分模型仍只能处理纯文本，图片发送前后都会受 provider 能力限制。
